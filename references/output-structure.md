# Output Structure Reference

Genericized from an approved leadership brief example. Client name, named individuals, and specific financial figures have been replaced with placeholders. The structure, tone, and table shapes are the pattern to follow — do not copy any client specifics from a past engagement into a new one.

## Header block

- Title: AI Operating Model Report
- Subtitle: Leadership Brief
- To: Leadership team
- Source: [Report type] | [Engagement ID] | [Industry] | [Engagement ID]
- Prepared: [date]
- Status: [e.g. Leadership review draft based on supplied discovery data]

## Section 1 — Executive Readout

A single callout box: 3-5 bullets, each one a leadership takeaway. Pattern:
- Where the opportunity concentrates (name the workflow clusters, not every workflow)
- Where accountability stays with named human roles vs where AI supports
- What baseline data is missing or low-confidence, stated plainly
- Any structural finding needing immediate attention (e.g. "all N technology roles require validation")

Followed by a "Leadership Decisions Needed" list — 3-5 bullets, each phrased as a decision, not a task. ("Confirm which priority workflows enter the pilot first," not "review workflows.")

## Section 2 — Priority Workflow Opportunities

One sentence stating how many workflows were identified and the most repeated automation pattern.

Table columns: Workflow | Impact area | Automation pattern | Benefit

Keep benefit descriptions outcome-focused (what improves), not feature-focused (what the tool does).

## Section 3 — Operating Accountability (current RACI)

One sentence: how many activities the RACI extract covers, and the most frequent accountable groups (by name of function/role, not necessarily named individuals unless the client wants named owners in the brief).

A callout box stating the common RACI implication: AI supports preparation, routing, and reporting; the named accountable owner keeps final decision authority.

Table columns: Activity | Responsible | Accountable

## Section 4 — KPI Model and Evidence Gaps

One sentence: how many KPIs total, how many are flagged as missing or low-confidence baselines.

Table columns: KPI | Baseline | Target

Where a baseline doesn't exist in client data, the Baseline column reads "Baseline requires validation" — never fabricated. Target column states the governed outcome, not a specific number, until the baseline is confirmed.

## Section 5 — Technology Role Validation

One sentence framing: this is a dependency map for operating-model design, not a confirmed stack recommendation.

Table columns: Tool | Current Role

## Section 6 — Recommended Next Steps

3-5 bullets, each an operational next step (e.g. "Run a baseline validation sprint covering X, Y, Z"), not a restatement of the findings.

## Extensions for the full transformation engine (added beyond this reference example)

The reference example above is the current-state half of the brief. The full FSOMA Transformation Brief Kit adds:

- Future State Roadmap (phased)
- Future State RACI (redesigned)
- Technology Recommendations (keep/replace/consolidate/validate, not just "requires validation")
- 2x2 Impact vs Effort chart (embedded image, numbered markers plus legend when workflow count is high)
- Commercial Model (four benefit categories: AI productivity, spend recovery, tooling rationalisation, revenue growth, each cross-checked top-down / bottom-up / client-stated)
- Risk Flags and Validation Needed
- Data Foundation Readiness (only when the optional Data Inventory input category is provided)

Every KPI, RACI, and technology table row carries an evidence-level tag: Validated, Partially validated, or Needs validation. Use the client's own confidence or validation-status field when provided, don't re-derive one from scratch.

Every RACI row missing Responsible or Accountable is flagged as a governance gap, in both current-state and future-state RACI sections, never filled with an inferred owner.
